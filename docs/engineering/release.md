# Release Process

## Versioning

The project follows Semantic Versioning once public releases begin. Until then, `0.0.x` can be used for internal bootstrap checkpoints.

## Release Inputs

Before cutting a release:

- update the changelog
- confirm milestone exit criteria are met
- verify the repository structure still matches architecture docs
- note any intentionally deferred risks

## Release Notes

Release notes should explain:

- what user-visible capability was added
- what remains explicitly out of scope
- any migration or configuration impact
