# Release 0.2.0 Checklist

## Purpose

This checklist is the execution and review baseline for `0.2.0`.

`0.2.0` is the stage where Code Orb should move from a narrow MVP proof into a small but real alpha for local repository work.

## Primary Benchmark

The first target for `0.2.0` is:

- [benchmarks/failing-test-fix/task.md](../../../benchmarks/failing-test-fix/task.md)

`0.2.0` should be executed as a benchmark-driven release:

- first make the failing-test-fix benchmark pass end to end
- then generalize carefully beyond that path

For implementation order, use [0.2.0-todo.md](./0.2.0-todo.md).

## Ordered Todo List

Follow this order unless there is a strong reason to change it:

1. make `orb run` runnable and understandable for the failing-test-fix benchmark
2. make the agent inspect benchmark failure output before editing
3. make the agent locate the relevant implementation from repository context
4. make the agent choose read/search/edit/verify actions based on bug-fix state rather than one replacement-only path
5. make the agent fix the benchmark and rerun verification
6. tighten reporting so the benchmark output is believable for a user review
7. add and pass additional task shapes only after the primary benchmark path is solid

## Failing-Test-Fix Benchmark Path

- [x] the agent can start from the benchmark task description and repository, not from a narrow replacement prompt
- [x] the agent can inspect failing verification output before editing
- [x] the agent can locate the relevant implementation file from repository context
- [x] the agent can make a minimal implementation fix
- [x] the agent can rerun benchmark verification successfully
- [x] the final report explains the bug, the fix, and the verification result

## Prompt Management

- [x] system, planner, executor, and reporter prompts exist as repository-managed assets
- [x] the benchmark path can rely on prompt assets rather than only hidden inline strings
- [x] prompt responsibilities are separated cleanly enough to evolve during `0.2.0`

## CLI Usability

- [x] the CLI has a clear runnable entry path for contributors
- [x] help, usage, and environment/setup errors are understandable
- [x] the CLI can run on a local repository outside the current fixture-only workflow

## Task Generalization

- [x] the agent can handle more than one narrow prompt pattern
- [x] task intake is not hard-coded to one replacement-oriented flow
- [x] the agent can support at least a small set of realistic task classes

Suggested initial task classes:

- documentation edit
- simple code replacement
- small verification-driven fix

## Repository Understanding

- [x] the runtime performs meaningful context gathering before editing
- [x] context gathering is reflected in plan and event flow
- [x] repository understanding is broader than one string search path

## Planning And Tool Selection

- [x] planning is less template-specific than the current MVP path
- [x] tool selection is driven by task and repository context rather than one hard-coded sequence
- [x] the runtime can decide between read/search/edit/verify actions based on task state

## Approval And Safety

- [x] mutating actions have understandable approval behavior
- [x] denial paths remain coherent and visible
- [x] the CLI interaction for approval is usable enough for alpha testing

## Reporting

- [x] the final report clearly explains what changed
- [x] the final report clearly explains what was validated
- [x] the final report clearly explains remaining risks or limitations

## Test Coverage

- [x] the failing-test-fix benchmark is covered by an end-to-end validation path
- [x] fixture repositories cover more than one narrow MVP scenario
- [x] e2e tests cover a small but real alpha task set
- [x] unit, integration, and e2e tests remain green together

## Scope Discipline

- [x] `0.2.0` does not depend on subagents, daemon mode, or desktop UI
- [x] `0.2.0` does not require advanced provider routing
- [x] the implementation remains focused on alpha usability and task generalization

## Exit Rule

`0.2.0` is complete only when Code Orb can be run as a believable alpha on local repositories for a small but real set of tasks, without relying on hidden operator knowledge.
