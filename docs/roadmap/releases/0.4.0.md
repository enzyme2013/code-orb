# Release 0.4.0

- Status: Released
- Release date: 2026-04-06

## Intent

Make Code Orb aware of prior runs and current repository state so it can explain work in context instead of acting like each run starts from a blank slate.

`0.4.0` should turn the current single-run agent into a local coding agent that can persist session results, inspect git working-tree state, and continue follow-up work with explicit context.

## In Scope

- persist session results as local inspectable artifacts
- add a minimal session store that can save, list, and load session records
- collect git working-tree state at runtime for repositories that are git-backed
- distinguish pre-existing repository changes from changes caused by the current run
- include repository-state context in final reporting
- support a minimal follow-up flow based on a prior saved session
- add milestone benchmarks for existing-changes awareness and session follow-up

## Out Of Scope

- automatic commit, branch, stash, or rebase workflows
- full execution checkpoint restore or resumable internal state machines
- remote session sync or shared session backends
- multi-repository orchestration
- pull request or GitHub workflow automation
- merge conflict resolution flows

## Exit Criteria

- each `orb run` persists a session artifact containing task, cwd, timestamps, outcome, summary, changed files, validations, and risks
- the runtime can list recent session artifacts and load a specific prior session record
- git-backed runs capture repository state clearly enough to report:
  - current branch when available
  - whether the repository was already dirty before the run
  - staged, unstaged, and untracked file paths at snapshot time
- final reporting distinguishes pre-existing repository changes from changes caused by the current run
- a follow-up run can consume prior session context well enough to reference the earlier outcome and continue work intentionally
- unit tests cover session persistence, git snapshot collection, change classification, and follow-up context building
- the milestone benchmarks pass:
  - `existing-changes-awareness`
  - `session-follow-up`

## Working Focus

`0.4.0` should favor repository awareness and continuity over broader task breadth.

The milestone question is:

- can Code Orb explain what it changed relative to the repository's prior state, and can it carry forward enough prior-session context to support a deliberate follow-up run?

## Validation Strategy

### Unit Test

- session artifact schema and serialization
- session store save, list, and load behavior
- git working-tree snapshot collection and parsing
- classification of existing repository changes versus current-run changes
- follow-up context construction from a prior session record

### Benchmark

- `existing-changes-awareness`
  - start from a repository that already has local uncommitted changes
  - run a small task
  - verify that final reporting distinguishes prior changes from current-run changes
- `session-follow-up`
  - run an initial task that creates a session artifact
  - run a second task that uses the prior session as context
  - verify that the second run reports follow-up behavior rather than acting like a blank-slate run

## Suggested Task Packages

### 1. Session Artifact Foundation

- define the session artifact shape
- choose the local storage location
- persist session results at the end of each run

### 2. Session Inspection Flow

- add session listing and lookup primitives
- support loading a prior session for follow-up context

### 3. Git State Snapshotting

- collect git repository state before and after runs when git is available
- make dirty-state and changed-file reporting explicit

### 4. Reporting In Repository Context

- show repository-state context in final output
- distinguish existing changes from current-run changes clearly

### 5. Benchmark And Test Coverage

- add unit coverage for session and git-awareness logic
- add the two milestone benchmarks and keep them stable under repeated runs
