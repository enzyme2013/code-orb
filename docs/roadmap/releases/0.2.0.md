# Release 0.2.0

- Status: Released
- Release date: 2026-04-06

## Intent

Ship the first alpha-quality local coding agent that is runnable on a real repository without being limited to the current narrow replacement-oriented MVP path.

The primary validation target for this release is:

- [benchmarks/failing-test-fix/task.md](../../../benchmarks/failing-test-fix/task.md)

## In Scope

- make the CLI runnable and understandable for real local-repository usage
- make the agent pass the `failing-test-fix` benchmark end to end
- establish a minimal prompt management baseline for system, planner, executor, and reporter prompts
- improve task intake so the agent can understand a failing-test-fix style request without depending on one replacement-only prompt shape
- improve repository understanding and context gathering before edits
- make planning and tool selection less task-template-specific, starting with the bug-fix benchmark path
- strengthen approval UX for mutating actions
- improve final reporting so users can understand what changed, what was validated, and what remains risky
- add more representative fixture repositories and end-to-end scenarios after the primary benchmark path is solid

## Out Of Scope

- multi-agent execution
- daemon mode
- dynamic skills and plugins
- advanced multi-provider routing
- desktop shell
- long-term memory systems

## Exit Criteria

- the CLI can be run by a contributor on a local repository with clear setup and error handling
- the agent can complete the `failing-test-fix` benchmark from prompt to report
- prompt assets are managed as repository artifacts rather than hidden string literals
- repository understanding and tool selection are no longer tied to one hard-coded replacement flow
- approval behavior is understandable for mutating actions
- fixture and e2e coverage reflects at least the primary benchmark path, with room to expand to a small alpha task set

## Working Focus

This release should prioritize usability and generalization over adding many new subsystems.

The core question for `0.2.0` is:

- can a technically capable user run Code Orb on a local repository and get useful behavior on a small set of real coding-agent tasks?

The first concrete proof for that question should be:

- can Code Orb diagnose and fix the bug in `benchmarks/failing-test-fix/repo/` without relying on a narrow replacement-only task path?

## Release Assessment

As of 2026-04-06, the repository-level `0.2.0` goals are satisfied:

- the failing-test-fix benchmark passes from natural-language task to final report
- the failing-test-fix benchmark runner now executes in a temporary copy so repeated benchmark runs do not mutate the source benchmark repository
- prompt assets exist and are used by the benchmark path
- CLI usability, approval behavior, reporting, and test coverage are at alpha level
- the current verification baseline passes:
  - `pnpm test:unit`
  - `pnpm test:integration`
  - `pnpm test:e2e`
