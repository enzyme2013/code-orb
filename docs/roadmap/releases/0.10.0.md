# Release 0.10.0

- Status: Planned

## Intent

Make workspace guidance and runtime configuration first-class parts of normal Code Orb use.

`0.10.0` is the milestone where Code Orb should stop relying on informal environment setup and ad hoc repository context for important operating guidance. Project instructions, configuration precedence, provider selection, and capability visibility should become explicit product surfaces.

## In Scope

- define and implement the workspace-instruction model beyond the minimal usable-CLI support from `0.8.0`
- support repository guidance such as `AGENTS.md` through a documented precedence model
- define how repository-level, user-level, and CLI/session-level configuration interact
- add provider profiles or similarly explicit configuration units instead of relying only on raw environment variables
- make active provider, model, and relevant capability boundaries visible enough for operators to understand the current runtime mode
- document the authoritative instruction and configuration surfaces in product and architecture docs

## Out Of Scope

- durable autonomous memory systems
- full skill ecosystems
- MCP or plugin ecosystems
- desktop shells
- generalized multi-agent orchestration

## Exit Criteria

- Code Orb has a documented instruction-loading and configuration-precedence model
- repository guidance such as `AGENTS.md` is handled predictably rather than as an implicit side channel
- provider and model selection can be managed through an explicit configuration surface
- operators can tell which instructions and provider settings are active for a run or session
- current CLI flows remain functional after the configuration and instruction model becomes more explicit

## Working Focus

`0.10.0` should make context and configuration legible, not magical.

The milestone question is:

- can Code Orb make project guidance and provider configuration explicit enough that users understand how the agent is being steered?

## Suggested Task Packages

### 1. Instruction Model

- define discovery and precedence for project guidance
- keep the source of truth obvious to users

### 2. Configuration Model

- define provider profiles, overrides, and precedence
- make runtime mode visible

### 3. Documentation And Reporting

- document the operating model clearly
- expose active instruction and configuration state in CLI flows where useful

## Validation Strategy

### Unit Test

- instruction discovery and precedence
- config precedence and override behavior
- provider profile resolution

### Integration Test

- repository-guided runs with explicit configuration
- conflicting config sources resolved predictably

### End-To-End Or Benchmark

- common CLI flows still work with the new instruction and configuration model
- at least one realistic task path depends on repository guidance being loaded correctly
