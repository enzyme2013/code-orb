# Release 0.10.0

- Status: Planned

## Intent

Make workspace guidance, agent identity, and runtime configuration first-class parts of normal Code Orb use.

`0.10.0` is the milestone where Code Orb should stop relying on informal environment setup, ad hoc repository context, and hidden persona prompting for important operating guidance. Project instructions, identity/profile precedence, provider selection, and capability visibility should become explicit product surfaces.

## In Scope

- define and implement the workspace-instruction model beyond the minimal usable-CLI support from `0.8.0`
- support repository guidance such as `AGENTS.md` through a documented precedence model
- add an explicit identity/profile surface inspired by `SOUL.md`-style assets without making it a hidden side channel
- define how repository-level guidance, identity/profile inputs, user-level configuration, and CLI/session-level configuration interact
- add provider profiles or similarly explicit configuration units instead of relying only on raw environment variables
- make active provider, model, identity/profile, and relevant capability boundaries visible enough for operators to understand the current runtime mode
- document the authoritative instruction, identity, and configuration surfaces in product and architecture docs

## Out Of Scope

- durable autonomous memory systems
- self-mutating identity profiles or hidden persona state
- full skill ecosystems
- MCP or plugin ecosystems
- desktop shells
- generalized multi-agent orchestration

## Exit Criteria

- Code Orb has a documented instruction-loading, identity-profile, and configuration-precedence model
- repository guidance such as `AGENTS.md` is handled predictably rather than as an implicit side channel
- any `SOUL.md`-style profile surface is explicit, inspectable, and subordinate to the documented precedence model rather than competing with repository rules silently
- provider and model selection can be managed through an explicit configuration surface
- operators can tell which instructions, identity/profile surfaces, and provider settings are active for a run or session
- current CLI flows remain functional after the configuration and instruction model becomes more explicit

## Working Focus

`0.10.0` should make context, identity, and configuration legible, not magical.

The milestone question is:

- can Code Orb make project guidance, agent identity, and provider configuration explicit enough that users understand how the agent is being steered?

## Suggested Task Packages

### 1. Instruction Model

- define discovery and precedence for project guidance
- keep the source of truth obvious to users

### 2. Identity Profile Model

- define `SOUL.md`-style profile loading, precedence, and visibility rules
- keep identity/profile inputs explicit instead of hidden prompt state

### 3. Configuration And Reporting

- define provider profiles, overrides, and precedence
- expose active instruction, identity/profile, and configuration state in CLI flows where useful

## Validation Strategy

### Unit Test

- instruction discovery and precedence
- identity/profile loading and precedence
- config precedence and override behavior
- provider profile resolution

### Integration Test

- repository-guided runs with explicit configuration
- repository-guided runs with explicit identity/profile inputs
- conflicting config sources resolved predictably

### End-To-End Or Benchmark

- common CLI flows still work with the new instruction and configuration model
- at least one realistic task path depends on repository guidance or identity/profile guidance being loaded correctly
