# Release 0.7.0

- Status: Released on 2026-04-08

## Intent

Turn the contracts clarified in `0.6.0` into an explicit session-backed query runtime and a more deliberate tool and provider runtime.

`0.7.0` is where Code Orb should stop looking like a set of task-specific execution branches inside one agent engine and start looking like a real runtime loop that can sustain multiple internal iterations inside one turn.

This is the milestone where Code Orb should align more closely with the useful runtime shape learned from the Claude Code reverse-engineering work in runtime layering, and with the Codex reverse-engineering work in provider transport and continuation robustness, without trying to match either product's full breadth.

## Why This Milestone Exists

`0.6.0` made provider compatibility, generated-edit execution, and tool registration more explicit, but the runtime still concentrated turn execution inside one engine through ordered task-specific paths.

Today, `BasicAgentEngine` still decides real work through specialized branches such as failing-test repair, replace-and-verify, and generated file writes, while tool execution and verification outcomes are only partially visible as reusable loop state.

At the same time, live-provider testing showed that a runtime loop alone is not enough: provider continuation and fallback behavior also need a deliberate architecture instead of ad hoc adapter assumptions.

`0.7.0` is where those tactical paths should stop defining architecture accidentally and start flowing through one explicit runtime skeleton.

## In Scope

- implement an explicit turn-level query loop that can perform more than one internal iteration before producing a final turn result
- make the boundary between session-owned state and single-turn loop execution explicit enough that later shells and richer runtimes do not depend on one monolithic engine
- define and implement explicit loop transitions for:
  - model interaction
  - deterministic runtime work
  - tool execution
  - verification
  - retry or repair
  - stop
- make tool results and verification results re-enter turn state through a documented runtime path instead of ad hoc task-specific branches
- evolve the tool runtime from a built-in dispatcher into a clearer orchestration layer with:
  - registry-backed tool lookup
  - validation before execution
  - declared execution semantics per tool
  - clearer error and metadata handling
- establish a provider adapter baseline that owns:
  - transport selection
  - response normalization
  - continuation state handling
  - fallback behavior when continuation or transport assumptions fail
- define the initial orchestration policy for read-only versus mutating tools, even if execution remains mostly serial
- make provider capability differences visible to the runtime where they affect loop behavior or tool availability
- update architecture docs to reflect the implemented loop and tool-runtime shape

## Out Of Scope

- full Claude Code parity
- full Codex transport stack parity
- streaming tool execution
- deferred tool search or dynamic large-tool-set discovery
- subagents, skills, MCP, or plugin ecosystems as runtime defaults
- background daemon mode
- desktop UI implementation
- broad multi-provider routing as a product feature

## Exit Criteria

- one turn may involve multiple explicit internal loop iterations without depending on hidden special-case branches
- session-owned runtime state and single-turn loop state are separated clearly enough that later shells or richer runtimes do not depend on one oversized engine object
- loop transitions and stop conditions are documented and reflected in implementation
- tool execution, verification, and retry behavior re-enter turn state through one coherent runtime path
- built-in tools are provided through a registry-backed runtime boundary rather than only through executor-local hardcoding
- the tool runtime has explicit execution semantics for mutating versus read-only behavior, even if advanced concurrency is still deferred
- provider capability differences that affect runtime behavior are surfaced through normalized runtime-facing contracts
- provider continuation and transport failures have a documented adapter-owned fallback policy instead of being treated as undefined runtime behavior
- existing CLI flows still work without regressing `orb run`, `orb chat`, session artifacts, or the current benchmarks

## Working Focus

`0.7.0` should align Code Orb with Claude Code at the runtime-architecture level and with Codex at the provider-adapter robustness level, not at the feature-count level.

The milestone question is:

- can Code Orb execute one user turn through an explicit reusable runtime skeleton, with a provider adapter that fails predictably instead of opaquely, without dragging in the heavier product systems of Claude Code or Codex?

## Suggested Task Packages

### 1. Query Loop Runtime

- implement the explicit turn loop
- define loop state, transitions, and stop reasons

### 2. Session Engine Boundary

- separate session-owned state from single-turn execution state
- avoid letting one engine object accidentally become the whole runtime

### 3. Tool Runtime

- introduce a real tool registry boundary
- separate validation, dispatch, and execution concerns

### 4. Verification And Repair Flow

- route verification and repair through the same loop semantics instead of bespoke task branches

### 5. Provider Adapter Baseline

- expose capability differences that matter to loop and tool behavior
- keep provider-specific quirks, transport policy, continuation state, and fallback behavior behind adapter contracts

### 6. Compatibility And Benchmarks

- preserve current CLI usability while replacing internal runtime structure
- keep milestone tests and benchmarks green

## Current Validation Status

Validation on 2026-04-08 covered:

- `pnpm build`
- `pnpm test`
- `pnpm run benchmark:interactive-multi-turn`
- `pnpm run benchmark:failing-test-fix`

This validation round confirmed:

- one turn now executes through explicit loop state with bounded stop reasons
- tool results can re-enter turn state through both deterministic verification flows and provider-capability-driven model follow-up
- the OpenAI Responses adapter now exposes runtime-owned tool definitions and normalizes native tool calls back into the shared model contract
- `orb run`, `orb chat`, session artifacts, CLI e2e coverage, and the current benchmarks remained green after the runtime refactor

Manual live-provider smoke testing on 2026-04-08 also found one OpenAI-compatible gateway limitation:

- the configured gateway can trigger tool calls and the runtime can execute them
- direct `previous_response_id` continuation was not sufficient on its own
- after landing adapter-owned full replay fallback that replays the assistant `function_call` together with `function_call_output`, the same live-provider smoke completed successfully

This confirms that:

- the turn loop can enter provider-native tool-calling mode
- provider robustness depends on adapter-owned fallback policy rather than the turn loop alone

## Release Validation Status

As of 2026-04-08, `0.7.0` is considered released because:

- turn execution now runs through an explicit reusable query loop with bounded stop reasons
- session-owned state, turn-loop state, tool orchestration, and provider runtime responsibilities are explicit in both code and architecture docs
- canonical tool-result handling is separated from provider-specific wire encoding
- the OpenAI-compatible adapter owns continuation and full-replay fallback behavior instead of relying on undefined transport assumptions
- `orb run`, `orb chat`, session artifacts, CLI e2e coverage, benchmark coverage, and the live provider smoke path all remained functional after the refactor
