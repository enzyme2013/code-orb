# Release 0.7.0

- Status: Planned

## Intent

Turn the contracts clarified in `0.6.0` into an explicit agent query loop and a more deliberate tool runtime.

`0.7.0` is where Code Orb should stop looking like a set of task-specific execution branches inside one agent engine and start looking like a real runtime loop that can sustain multiple internal iterations inside one turn.

This is the milestone where Code Orb should align more closely with the useful runtime shape learned from the Claude Code reverse-engineering work, without trying to match Claude Code's full product breadth.

## In Scope

- implement an explicit turn-level query loop that can perform more than one internal iteration before producing a final turn result
- define and implement explicit loop transitions for:
  - model interaction
  - deterministic runtime work
  - tool execution
  - verification
  - retry or repair
  - stop
- make tool results and verification results re-enter turn state through a documented runtime path instead of ad hoc task-specific branches
- evolve the tool runtime from a built-in dispatcher into a clearer orchestration layer with:
  - registry-backed tool lookup
  - validation before execution
  - declared execution semantics per tool
  - clearer error and metadata handling
- define the initial orchestration policy for read-only versus mutating tools, even if execution remains mostly serial
- make provider capability differences visible to the runtime where they affect loop behavior or tool availability
- update architecture docs to reflect the implemented loop and tool-runtime shape

## Out Of Scope

- full Claude Code parity
- streaming tool execution
- deferred tool search or dynamic large-tool-set discovery
- subagents, skills, MCP, or plugin ecosystems as runtime defaults
- background daemon mode
- desktop UI implementation
- broad multi-provider routing as a product feature

## Exit Criteria

- one turn may involve multiple explicit internal loop iterations without depending on hidden special-case branches
- loop transitions and stop conditions are documented and reflected in implementation
- tool execution, verification, and retry behavior re-enter turn state through one coherent runtime path
- built-in tools are provided through a registry-backed runtime boundary rather than only through executor-local hardcoding
- the tool runtime has explicit execution semantics for mutating versus read-only behavior, even if advanced concurrency is still deferred
- provider capability differences that affect runtime behavior are surfaced through normalized runtime-facing contracts
- existing CLI flows still work without regressing `orb run`, `orb chat`, session artifacts, or the current benchmarks

## Working Focus

`0.7.0` should align Code Orb with Claude Code at the runtime-architecture level, not at the feature-count level.

The milestone question is:

- can Code Orb execute one user turn through an explicit reusable query loop and tool runtime without dragging in Claude Code's heavier product systems?

## Suggested Task Packages

### 1. Query Loop Runtime

- implement the explicit turn loop
- define loop state, transitions, and stop reasons

### 2. Tool Runtime

- introduce a real tool registry boundary
- separate validation, dispatch, and execution concerns

### 3. Verification And Repair Flow

- route verification and repair through the same loop semantics instead of bespoke task branches

### 4. Provider Capability Wiring

- expose capability differences that matter to loop and tool behavior
- keep provider-specific quirks behind adapter contracts

### 5. Compatibility And Benchmarks

- preserve current CLI usability while replacing internal runtime structure
- keep milestone tests and benchmarks green

## Validation Strategy

### Unit Test

- loop state transitions
- stop and retry conditions
- tool registry and validation behavior
- provider capability mapping

### Integration Test

- one turn with multiple internal iterations
- tool execution followed by verification and follow-up loop continuation
- mutating and read-only tool behavior through the shared runtime path

### End-To-End Or Benchmark

- existing CLI benchmark coverage still passes
- at least one milestone path proves a multi-iteration turn in real runtime flow
