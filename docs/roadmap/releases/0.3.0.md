# Release 0.3.0

- Status: Released
- Release date: 2026-04-06

## Intent

Strengthen Code Orb's edit-and-verify loop so it behaves more like a believable coding agent when tasks do not succeed on the first try.

`0.3.0` should make the existing alpha task set more reliable before the project expands into broader workflow or session features.

## In Scope

- make patch application and edit execution more robust
- classify edit failures so they can be handled and reported clearly
- strengthen verification loops after edits, including clearer retry boundaries
- improve failure handling when a verification-driven repair does not succeed
- improve final reporting so successful and failed runs are both reviewable
- add automated coverage for edit failure and verify-still-failed paths

## Out Of Scope

- new shell surfaces such as desktop
- session persistence or resume flows
- git-aware diff, commit, or branch workflows
- multi-agent execution
- plugin or skill extension systems
- broad new benchmark families unrelated to edit-and-verify reliability

## Exit Criteria

- edit failures are classified clearly enough to distinguish target-not-found, tool failure, and approval denial paths
- verification-driven tasks can distinguish:
  - verification already passed
  - verification failed but could not be diagnosed
  - a repair was attempted but no additional retry was available
  - a repair was applied but verification still failed
  - a repair was applied and verification passed
- final reports explain changed files, attempted validation, stop reason, and remaining risk in both success and failure cases
- the existing `replace-string` and `failing-test-fix` end-to-end paths remain green
- at least one additional failure-oriented end-to-end scenario is covered by tests

## Release Assessment

As of 2026-04-06, the repository-level `0.3.0` goals are satisfied:

- edit failures are classified across target-not-found, path safety, and approval denial paths
- verification-driven runs distinguish successful repair, post-edit verification failure, and blocked edit outcomes
- terminal output now reports explicit turn status and session outcome alongside changed files, validations, and risks
- benchmark coverage now spans three milestone-aligned result classes:
  - `benchmarks/failing-test-fix/`
  - `benchmarks/verification-still-fails/`
  - `benchmarks/approval-denied/`
- the current verification baseline passes:
  - `pnpm test:unit`
  - `pnpm test:integration`
  - `pnpm test:e2e`
  - `pnpm exec tsc --noEmit`
  - `pnpm run benchmark:failing-test-fix`
  - `pnpm run benchmark:verification-still-fails`
  - `pnpm run benchmark:approval-denied`

## Working Focus

`0.3.0` should favor reliability over breadth.

The milestone question is:

- can Code Orb make and verify small repository edits in a way that remains understandable even when the first edit or repair attempt fails?

## Execution Order

1. define structured edit failure categories in the tool and engine path
2. make verification loops report clearer repair outcomes and retry boundaries
3. improve final reporting and terminal rendering for failure cases
4. add unit, integration, and e2e coverage for failure-oriented edit-and-verify scenarios

## Suggested Task Packages

### 1. Edit Execution Hardening

- preserve the current minimal tool set, but make edit failures explicit
- ensure edit errors can be surfaced without collapsing into one generic runtime failure

### 2. Verification Loop Clarity

- make verify-after-edit behavior easier to follow in runtime events and reports
- keep retry logic bounded and explain when the runtime stops retrying

### 3. Report Improvement

- make final output useful for user review when a task succeeds
- make final output equally useful when a task stops on an edit or verification failure

### 4. Test Matrix Expansion

- cover at least one edit failure path
- cover at least one verification-still-failed path
- keep the current success paths green
