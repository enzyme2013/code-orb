# Release 0.11.0

- Status: Planned

## Intent

Add explicit memory and reusable skills without turning either one into opaque magic or hidden external state.

`0.11.0` is the milestone where Code Orb should improve continuity across sessions and support reusable guided workflows, while keeping both memory and skills inspectable, bounded, and consistent with the core runtime model.

## In Scope

- define an explicit memory model beyond the current session-artifact and follow-up flow
- support inspectable memory scopes such as:
  - repository or workspace memory
  - user preference memory
  - task or workflow carry-forward where it is explicit and reviewable
- define when memory is read, when it may be written, and how users can inspect or correct it
- define a memory-provider boundary so optional external memory backends can contribute only through explicit, inspectable records with provenance
- add a first-class skill model for reusable prompt or workflow assets
- support project or user skills without collapsing skills into the tool layer
- keep skill invocation and memory effects aligned with the existing safety and event model

## Out Of Scope

- opaque self-mutating memory systems with unclear provenance
- provider-managed or externally hosted memory that bypasses the explicit memory contract, event model, or reporting flow
- a broad plugin marketplace
- MCP as a required dependency for skills
- generalized multi-agent orchestration
- desktop shells

## Exit Criteria

- Code Orb has an explicit, inspectable memory model that improves continuity beyond the current narrow follow-up path
- users can tell what memory was used and where it came from
- if optional external memory is used, users can still inspect the normalized memory records and their provenance instead of receiving invisible prompt injection
- project or user skills can be defined and invoked through a documented model
- skills remain distinct from tools in runtime semantics and documentation
- memory and skills do not bypass the existing safety, event, or reporting model

## Working Focus

`0.11.0` should improve continuity and repeatability without making the runtime harder to reason about, even when memory originates outside the local session store.

The milestone question is:

- can Code Orb remember and reuse useful context, including optional external memory inputs, while keeping memory and skills explicit enough to inspect, trust, and debug?

## Suggested Task Packages

### 1. Memory Contract

- define scopes, provenance, and lifecycle for explicit memory
- define inspection and correction surfaces
- define normalization and rejection rules for optional external memory sources

### 2. Skill Model

- define skill assets, loading rules, and invocation semantics
- keep skills distinct from tools

### 3. Runtime Integration

- integrate memory and skills with events, safety, and reporting
- avoid hidden side channels

## Validation Strategy

### Unit Test

- memory scope and provenance behavior
- external-memory normalization and rejection behavior
- skill discovery and invocation rules
- memory and skill event emission

### Integration Test

- repeated guided workflows across sessions
- local and optional external memory sources coexist under one inspectable contract
- project or user skills affecting runtime behavior predictably

### End-To-End Or Benchmark

- realistic tasks benefit from explicit memory continuity
- realistic tasks benefit from reusable skills without regressing core CLI behavior
- optional external memory paths do not regress auditability or runtime coherence
