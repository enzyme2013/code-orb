# Release 0.9.0

- Status: In Progress

## Intent

Improve the interactive CLI experience after Code Orb is already usable as a real tool.

`0.9.0` is the milestone where Code Orb should invest in terminal ergonomics, TUI-oriented interaction patterns, and operator efficiency without redefining the core runtime semantics again.

The purpose of this milestone is not to make the CLI prettier in isolation. It is to make sustained terminal-based coding work easier to drive, inspect, and trust.

## In Scope

- improve the interactive terminal UX for `orb chat`
- add clearer rendering for progress, plan state, tool activity, verification, and final outcomes
- improve session inspection and navigation inside the CLI so users can understand ongoing and completed work more quickly
- introduce TUI-oriented affordances where they materially improve usability:
  - better status surfaces
  - clearer approval prompts
  - richer diff or validation presentation
  - easier session and turn inspection
- preserve a clean shell/core boundary so terminal UX remains shell-owned
- keep the interactive CLI scriptable enough for automated testing where practical

## Out Of Scope

- a desktop application or desktop shell implementation
- a full plugin or command marketplace
- large new runtime semantics that should have landed in earlier milestones
- background daemon mode
- feature breadth expansions that mainly belong to product capability milestones rather than CLI UX
- broader instruction, identity, and configuration surfaces that belong to `0.10.0`

## Exit Criteria

- the interactive CLI presents progress, tool activity, verification, and outcomes more clearly than the minimal `0.5.x` shell
- users can inspect session state and recent work efficiently during longer interactive sessions
- approval and mutating-action UX are materially easier to review in terminal flow
- TUI-oriented improvements remain shell-owned and do not leak terminal-specific assumptions back into core runtime semantics
- existing CLI capabilities remain functional after the UI and interaction improvements

## Working Focus

`0.9.0` should optimize operator experience, not reopen core runtime architecture.

The milestone question is:

- once Code Orb is already usable, can the terminal experience become efficient and pleasant enough for sustained interactive work?

## Kickoff Notes

## Terminal UX Baseline Audit

Audit on 2026-04-09 confirmed:

- the current interactive shell already renders runtime-derived events instead of inventing separate execution semantics in the CLI
- `TerminalEventRenderer` is still a mostly flat line-oriented event stream, so plan state, tool activity, verification, and risk output are all present but not yet grouped or prioritized for scanability
- `/status` currently exposes only session id, mode, turn count, session status, task, and last-turn summary
- `orb sessions list` and `orb sessions show` already expose saved artifacts, mutating actions, validations, and repository-state notes, but only as sequential text with limited navigation or turn-level inspection affordances
- approval confirmation currently depends on the runtime-provided summary plus a simple `[y/N]` prompt, so richer target, diff, and consequence review remains shell-UX work for `0.9.0`

## Shell-Owned State Model At Kickoff

At kickoff, `0.9.0` should treat the shell-owned UX layer this way:

- progress and status surfaces should be derived from runtime events such as `session.started`, `turn.started`, `step.started`, `plan.generated`, `tool.*`, `verify.*`, `turn.completed`, and `session.completed`
- live inspection should derive from in-memory session runtime state already available to the interactive shell
- post-hoc inspection should derive from persisted session reports and artifacts instead of shell-local shadow state
- approval UX should remain a rendering and interaction layer over runtime approval requests, approval responses, and mutating-action reports
- no new ADR or architecture-doc change is required at kickoff because the existing shell/core boundary already covers this ownership model

## Acceptance Findings

- summary: the real CLI entrypoint could silently no-op when launched through `node --import tsx apps/cli/src/main.ts ...` on Windows because the direct-execution guard built an invalid `file://` URL from `process.argv[1]`
- found during: manual CLI closeout validation on 2026-04-11
- affected flow/test/benchmark: explicit `orb run`, `orb sessions list`, and `orb sessions show` command-line execution outside the in-process test harness
- scope relation: Current Release Closeout Fix
- decision: fixed in `0.9.0` by normalizing the entrypoint path with `pathToFileURL` and by adding regression coverage for the real CLI entrypoint path
- target version: `0.9.0`
- release blocking: yes

Validation still not covered:

- manual TTY inspection of the richer approval prompt presentation
- longer interactive shell smoke coverage outside the automated harness

## Release Validation Status

`0.9.0` is not yet considered released because:

- the real TTY manual execution coverage promised for primary `orb chat` sessions has not been completed yet
- the remaining open validation is about release honesty, not about newly discovered automated regressions in the current shell implementation
