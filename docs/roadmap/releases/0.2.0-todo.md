# Release 0.2.0 Todo List

## Purpose

This document is the ordered implementation plan for `0.2.0`.

It is intentionally narrower than the full checklist. The rule is:

- finish the primary benchmark path first
- only then broaden to additional task shapes

Primary benchmark:

- [benchmarks/failing-test-fix/task.md](../../../benchmarks/failing-test-fix/task.md)

## Stage 1: Runnable Alpha Setup

- [x] Make the CLI runnable by a contributor without hidden setup knowledge.
- [x] Improve help, usage, and provider configuration error messages.
- [x] Make the benchmark invocation path explicit in docs and CLI help.

Exit condition:

- a contributor can understand how to run Code Orb and how to target the benchmark repo

## Stage 2: Benchmark-Aware Intake

- [x] Accept a natural-language failing-test-fix request without requiring the current replacement-oriented phrasing.
- [x] Treat the benchmark task description as the source of truth for expected behavior.
- [x] Ensure the runtime starts by inspecting verification output before editing code.

Exit condition:

- the agent begins from the benchmark task and failure signal, not from a hard-coded replacement path

## Stage 3: Prompt Management Baseline

- [x] Move system, planner, executor, and reporter guidance into prompt assets under `prompts/`.
- [x] Define clear prompt responsibilities for the failing-test-fix benchmark path.
- [x] Keep prompt assembly minimal and versioned instead of introducing a large prompt framework.

Exit condition:

- the benchmark path no longer depends on ad hoc prompt strings hidden in code

## Stage 4: Context Gathering For Bug Fixes

- [x] Read failing verification output and identify the relevant test file.
- [x] Read the relevant implementation file or files before proposing edits.
- [x] Reflect context gathering in plan items and runtime events.

Exit condition:

- the agent can explain why it is editing a file, based on evidence from tests and code

## Stage 5: Planning And Tool Selection

- [x] Make planning less template-specific than the current MVP path.
- [x] Make tool selection depend on task state and repository evidence.
- [x] Avoid one fixed `search -> replace -> verify` sequence for every task.

Exit condition:

- tool choice is driven by repository state, not only by a single narrow prompt pattern

## Stage 6: Fix And Verify Loop

- [x] Apply a minimal implementation fix for the benchmark bug.
- [x] Rerun the benchmark verification command.
- [x] If verification still fails, support at least one repair retry loop.

Exit condition:

- the failing-test-fix benchmark can pass end to end

## Stage 7: Alpha-Quality Approval And Reporting

- [x] Make approval prompts understandable for mutating actions.
- [x] Make denial behavior clear to the user.
- [x] Improve the final report so it explains the bug, the fix, verification, and remaining risks.

Exit condition:

- a technically capable user can understand what the agent did and why

## Stage 8: Broaden Beyond The Primary Benchmark

- [x] Add at least one additional task shape beyond failing-test-fix.
- [x] Add fixture and e2e coverage for the expanded alpha task set.
- [x] Keep unit, integration, and e2e verification green while broadening behavior.

Suggested next task shapes:

- documentation edit
- simple code replacement or rename
- small verification-driven fix in a second benchmark repo

Exit condition:

- `0.2.0` is no longer validated by only one narrow benchmark path

## Rule Of Order

Do not skip to Stage 8 before Stages 1 through 7 are stable.
