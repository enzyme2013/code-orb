# Release 0.12.0

- Status: Planned

## Intent

Define a deliberate extension boundary for external tools after the core runtime, CLI, configuration model, memory model, and skill model are already in place.

`0.12.0` is the milestone where Code Orb should open the door to MCP and plugin-style integrations without turning extensibility into the primary product before the foundations are ready.

## In Scope

- define the runtime boundary for external tool providers
- add an initial MCP integration path if it can conform to the core tool, event, and safety contracts
- define plugin-style registration boundaries where they materially differ from built-in or MCP-provided tools
- keep extension loading, permission handling, and auditability explicit
- document compatibility and support boundaries for external extensions

## Out Of Scope

- a plugin marketplace or broad discovery ecosystem
- uncontrolled third-party extension loading as the default product path
- generalized multi-agent orchestration
- desktop shells
- feature work that bypasses the core tool runtime model

## Exit Criteria

- Code Orb has a documented and implemented boundary for external tool integration
- MCP or plugin-style extensions that are supported flow through the same safety and event contracts as built-in tools
- extension loading and execution are explicit enough to audit and debug
- the runtime does not need to special-case external tools in a way that undermines its core contracts

## Working Focus

`0.12.0` should make extensibility deliberate, not accidental.

The milestone question is:

- can Code Orb support external tools without giving up the safety, auditability, and runtime coherence established by earlier milestones?

## Suggested Task Packages

### 1. Extension Boundary

- define registration and lifecycle for external tools
- keep the core runtime in control of execution semantics

### 2. MCP Integration

- support the minimum useful MCP path if it fits the core contracts
- document unsupported or degraded cases explicitly

### 3. Safety And Auditability

- preserve approval, event, and reporting behavior for external tools

## Validation Strategy

### Unit Test

- extension registration
- extension validation and metadata handling
- unsupported-extension failure paths

### Integration Test

- built-in and external tools coexist under one runtime model
- extension calls respect safety and reporting expectations

### End-To-End Or Benchmark

- at least one realistic extension path works without regressing core CLI behavior
