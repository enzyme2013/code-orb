# Release 0.4.0 Checklist

## Purpose

This checklist is the execution and review baseline for `0.4.0`.

`0.4.0` is the milestone where Code Orb should stop behaving like a blank-slate one-shot run and start carrying forward enough local state to explain repository context and support intentional follow-up work.

## Release Focus

The first priority for `0.4.0` is:

- persist an inspectable session artifact for every run
- capture git working-tree context before and after the run when git is available
- distinguish pre-existing repository changes from current-run changes in final reporting

For implementation order, use the priority plan below.

## Priority Plan

### P0: Contracts And Persistence Baseline

Goal:

- make every `orb run` leave behind a stable session record with enough data to support later reporting and follow-up work

Dependencies:

- none

Tasks:

- [x] define a stable `SessionArtifact` shape for serialized session records
- [x] extend report contracts with repository-state and session-artifact references where needed
- [x] choose and document the local artifact storage location
- [x] implement a minimal session store with `save`, `list`, and `load`
- [x] persist a session artifact at the end of each run

Suggested files:

- `packages/schemas/src/report.ts`
- `packages/schemas/src/runtime.ts`
- `packages/schemas/src/session-artifact.ts`
- `packages/schemas/src/index.ts`
- `packages/core/src/session/basic-session-runner.ts`
- `packages/core/src/session/session-store.ts`
- `packages/core/src/index.ts`
- `apps/cli/src/commands/run.ts`

Acceptance criteria:

- each `orb run` writes one inspectable artifact to the chosen local store
- the artifact contains task, cwd, timestamps, outcome, summary, changed files, validations, and risks
- the runtime can list recent artifacts and load a specific artifact by id without relying on in-memory state
- artifact serialization is covered by unit tests

### P0: Git Working-Tree Snapshotting

Goal:

- make the runtime aware of repository state before and after a run when the cwd is git-backed

Dependencies:

- depends on `P0: Contracts And Persistence Baseline`

Tasks:

- [x] define a normalized git snapshot shape for runtime and artifact use
- [x] add a git adapter that detects whether the cwd is a git repository
- [x] capture branch, dirty state, staged files, unstaged files, and untracked files before the run
- [x] capture the same git snapshot after the run
- [x] persist both snapshots in the session artifact when available

Suggested files:

- `packages/schemas/src/session-artifact.ts`
- `packages/schemas/src/report.ts`
- `packages/core/src/adapters/git/git-state-reader.ts`
- `packages/core/src/session/basic-session-runner.ts`
- `packages/core/src/index.ts`

Acceptance criteria:

- non-git repositories are handled explicitly without failing the run
- git-backed runs capture branch, dirty state, staged files, unstaged files, and untracked files before and after execution
- snapshot parsing and collection are covered by unit tests

### P1: Change Classification And Repository-Aware Reporting

Goal:

- make final reporting explain repository context honestly instead of attributing all dirty files to the current run

Dependencies:

- depends on `P0: Contracts And Persistence Baseline`
- depends on `P0: Git Working-Tree Snapshotting`

Tasks:

- [x] define change-classification output for pre-existing changes, current-run changes, and overlap cases
- [x] classify repository changes using before and after snapshots
- [x] include repository-state context in the final report contract
- [x] render repository-state context clearly in terminal output
- [x] surface artifact location in final reporting

Suggested files:

- `packages/schemas/src/report.ts`
- `packages/core/src/session/basic-session-runner.ts`
- `packages/core/src/session/change-classifier.ts`
- `apps/cli/src/rendering/terminal-event-renderer.ts`

Acceptance criteria:

- final reporting shows whether the repository was already dirty before the run
- final reporting distinguishes pre-existing changes from current-run changes
- files touched during the current run but already dirty before the run are represented explicitly instead of being hidden
- change-classification behavior is covered by unit tests

### P1: Follow-Up Session Context

Goal:

- allow a new run to intentionally reference a prior session record without restoring internal execution state

Dependencies:

- depends on `P0: Contracts And Persistence Baseline`
- depends on `P1: Session Inspection CLI Flow`

Tasks:

- [x] add a CLI option for starting a run from a prior session id
- [x] load the prior session artifact through the session store
- [x] construct a minimal follow-up context from prior task, outcome, summary, changed files, validations, and risks
- [x] inject follow-up context into runtime execution without pretending to restore checkpoints
- [x] record follow-up linkage in the new session artifact and final report

Suggested files:

- `apps/cli/src/commands/run.ts`
- `apps/cli/src/runtime/create-cli-runtime.ts`
- `packages/schemas/src/runtime.ts`
- `packages/schemas/src/report.ts`
- `packages/schemas/src/session-artifact.ts`
- `packages/core/src/session/basic-session-runner.ts`

Acceptance criteria:

- a second run can load a prior session artifact by id
- runtime context for the new run includes a deliberate prior-session summary instead of acting like a blank-slate run
- the resulting artifact and final report reference the prior session id
- follow-up context construction is covered by unit or integration tests

## Scope Discipline

- [x] `0.4.0` does not introduce automatic commit, branch, stash, or rebase flows
- [x] `0.4.0` does not claim resumable internal state-machine restoration
- [x] `0.4.0` does not introduce remote session sync or shared backends
- [x] `0.4.0` remains focused on local repository continuity and git-aware reporting

## Exit Rule

`0.4.0` is complete only when Code Orb can explain what it changed relative to the repository's prior state and can intentionally continue follow-up work using a saved local session record.
