# Release 0.5.0

- Status: Implemented with post-release usability fixes on 2026-04-08

## Intent

Turn Code Orb from a one-shot foreground command into a usable interactive CLI coding session for iterative local-repository work.

`0.5.0` should let a user stay inside one foreground session, issue multiple turns, inspect progress, and exit intentionally without having to restart the CLI for every follow-up request.

This milestone is where Code Orb should begin to feel like a local coding terminal rather than only a single-shot task runner.

## Status Clarification

`0.5.0` shipped the interactive shell, multi-turn session flow, and session artifact model on 2026-04-06.

After release, real usage exposed two gaps that were still part of `0.5` usability rather than `0.6` extensibility work:

- some OpenAI-compatible gateways returned empty non-streaming assistant content, so interactive chat could lose the assistant reply entirely
- general single-file coding requests inside `orb chat` could still stop at conversational output instead of applying the generated file write

Those gaps were closed on 2026-04-08 and should be treated as `0.5.x` stabilization work because they directly affected whether the interactive CLI was usable for small iterative coding tasks.

## In Scope

- add a dedicated interactive foreground command, expected to be `orb chat`
- keep one local session alive across multiple user turns within the same process
- treat each user input as a new turn inside the active session
- preserve session-scoped context across turns well enough for explicit follow-up requests inside the same session
- support a minimal interactive command surface for session usability:
  - `/help`
  - `/status`
  - `/exit`
- render turn results incrementally while keeping the shell interaction understandable in a plain terminal
- persist one inspectable session artifact for the interactive session with all turn reports
- keep the existing one-shot `orb run` path working alongside the new interactive flow
- add automated coverage for multi-turn interactive execution
- add a milestone benchmark for an interactive multi-turn coding session

## Out Of Scope

- background daemon mode or client/server session hosting
- a full-screen TUI, pane layout, or history browser
- simultaneous multi-session orchestration inside one terminal process
- full restoration of an interactive transcript from a saved artifact
- remote/shared session backends
- slash-command plugins or a broad command ecosystem beyond the minimal interactive control surface
- a fully general multi-file coding runtime or a broad provider capability matrix
- desktop UI work

## Exit Criteria

- a contributor can start an interactive coding session with the dedicated interactive CLI command
- without restarting the process, a user can complete at least two turns in the same session and receive separate turn reports
- follow-up requests inside the same session can reference earlier work intentionally rather than acting like unrelated blank-slate runs
- `/help`, `/status`, and `/exit` work inside the interactive session
- the session can close cleanly through an explicit exit command and through standard terminal end-of-input behavior
- the resulting session artifact records the session metadata plus multiple turn reports in one saved artifact
- final session reporting remains understandable after multiple turns
- the existing `orb run`, `orb sessions list`, and `orb sessions show` flows remain functional
- automated tests cover interactive command parsing, multi-turn session flow, session artifact persistence, and exit behavior
- the milestone benchmark passes:
  - `interactive-multi-turn`

## Working Focus

`0.5.0` should favor interaction quality and session continuity over visual polish or broad new backend abstractions.

The milestone question is:

- can a user stay inside one Code Orb CLI session and iteratively drive small coding tasks without re-invoking the CLI for each follow-up step?

## Validation Strategy

### Unit Test

- interactive command parsing for `orb chat`
- parsing and handling of `/help`, `/status`, and `/exit`
- session artifact accumulation across multiple turns
- exit and end-of-input handling helpers

### Integration Test

- scripted multi-turn interactive session using test IO
- session state continuity across at least two turns
- final report and artifact generation after a clean interactive exit

### End-To-End Test

- start an interactive session against a fixture repository
- complete one small edit-and-verify turn
- complete one follow-up turn in the same session
- verify repository output, turn rendering, and saved artifact structure

### Benchmark

- `interactive-multi-turn`
  - start an interactive session in a benchmark repository
  - complete one turn that makes or verifies a small repository change
  - complete a second turn that explicitly follows up on the first turn inside the same session
  - verify that one session artifact contains both turns and that the final session report remains coherent

### Manual Smoke Test

- run `orb chat` in a real TTY
- use `/help`
- complete at least two turns
- inspect `/status`
- exit with `/exit`

## Suggested Task Packages

### 1. Interactive Shell Loop

- add the foreground prompt/read/evaluate/render loop for `orb chat`
- keep the loop scriptable in tests without requiring a full TTY harness

### 2. Multi-Turn Session Runtime

- extend the session runner from one-turn execution to repeated turns within one session
- preserve turn boundaries and per-turn reporting inside the shared session

### 3. Session Control Commands

- implement `/help`, `/status`, and `/exit`
- keep the command surface intentionally small for the first interactive release

### 4. Session Artifact And Reporting

- persist one artifact per interactive session rather than one artifact per turn
- make the final session summary readable after multiple turns

### 5. Test And Benchmark Coverage

- add scripted tests for interactive flow control
- add the milestone benchmark and keep it stable under repeated runs

## Implementation Status

- `orb chat` starts a foreground interactive coding session
- one interactive session can execute multiple turns before exit
- `/help`, `/status`, and `/exit` are supported inside interactive mode
- interactive sessions persist one artifact containing all turn reports
- same-session follow-up context is injected from the prior turn without requiring `--from-session`
- automated coverage now includes unit, integration, and e2e tests for interactive mode
- milestone benchmark coverage now includes `interactive-multi-turn`
- interactive chat now renders assistant replies even when provider recovery needs a streaming fallback
- basic single-file generated writes inside `orb chat` now execute instead of stopping at conversational output
