# Release 0.5.0 Checklist

## Purpose

This checklist is the execution and review baseline for `0.5.0`.

`0.5.0` is the milestone where Code Orb should stop feeling like only a one-shot task command and start behaving like a foreground interactive coding session that supports multiple turns in one CLI process.

## Release Focus

The first priority for `0.5.0` is:

- start and keep alive one interactive CLI session in the foreground
- support multiple user turns inside that session
- make session control explicit with a minimal command surface

For implementation order, use the priority plan below.

## Priority Plan

### P0: Interactive CLI Contract And Command Surface

Goal:

- define the interactive shell entry point and the minimum command surface for session control

Dependencies:

- none

Tasks:

- [x] define the CLI contract for starting an interactive session, expected to be `orb chat`
- [x] define the interactive input model for normal user turns versus slash-style control commands
- [x] implement `/help`, `/status`, and `/exit` handling
- [x] keep help and usage text aligned with the new interactive command
- [x] keep the one-shot `orb run` and session inspection commands intact

Suggested files:

- `apps/cli/src/commands/run.ts`
- `apps/cli/src/main.ts`
- `apps/cli/src/runtime/create-cli-runtime.ts`
- `apps/cli/src/rendering/terminal-event-renderer.ts`
- `tests/unit/cli/run-command.test.ts`

Acceptance criteria:

- a contributor can start interactive mode from the CLI with a documented command
- `/help`, `/status`, and `/exit` are recognized inside interactive mode
- existing non-interactive CLI commands continue to parse and run correctly

### P0: Multi-Turn Session Runtime

Goal:

- allow one foreground session to own multiple user turns without creating a new process or blank-slate session for each request

Dependencies:

- depends on `P0: Interactive CLI Contract And Command Surface`

Tasks:

- [x] extend session execution from one-turn-only behavior to repeated turns in the same session
- [x] preserve explicit turn boundaries, per-turn planning, and per-turn reporting inside the shared session
- [x] make follow-up requests inside the same session reuse session-scoped context intentionally
- [x] keep event emission understandable for multi-turn execution
- [x] avoid breaking the existing one-shot session path used by `orb run`

Suggested files:

- `packages/core/src/session/basic-session-runner.ts`
- `packages/core/src/session/session-runner.ts`
- `packages/schemas/src/runtime.ts`
- `packages/schemas/src/report.ts`
- `packages/schemas/src/events.ts`
- `tests/integration/core/`

Acceptance criteria:

- one interactive session can complete at least two user turns in sequence
- turn reports remain distinct within the shared session
- same-session follow-up work does not behave like unrelated blank-slate runs
- one-shot execution remains supported

### P1: Interactive Rendering And Session Status

Goal:

- make the interactive shell understandable while the session remains open across multiple turns

Dependencies:

- depends on `P0: Interactive CLI Contract And Command Surface`
- depends on `P0: Multi-Turn Session Runtime`

Tasks:

- [x] render a readable interactive prompt cycle for user turns and runtime progress
- [x] render per-turn completion output without making the session-ending summary ambiguous
- [x] implement `/status` output that exposes the active session id, turn count, and current repository/session context at a useful minimal level
- [x] keep interactive rendering plain-terminal friendly instead of requiring a full-screen UI

Suggested files:

- `apps/cli/src/main.ts`
- `apps/cli/src/rendering/terminal-event-renderer.ts`
- `tests/unit/cli/`
- `tests/e2e/cli/`

Acceptance criteria:

- a user can understand when a turn starts, completes, and whether the session is still active
- `/status` gives a useful snapshot of the active session
- the interactive flow stays readable in a plain terminal transcript

### P1: Session Artifact And Exit Semantics

Goal:

- make interactive sessions auditable and closable without losing the multi-turn session record

Dependencies:

- depends on `P0: Multi-Turn Session Runtime`

Tasks:

- [x] define how one interactive session is persisted as one artifact with multiple turn reports
- [x] persist the interactive session artifact when the session exits cleanly
- [x] define behavior for end-of-input and explicit `/exit`
- [x] ensure final session reporting remains understandable after multiple turns
- [x] keep session inspection compatible with artifacts produced by both `orb run` and interactive mode

Suggested files:

- `packages/schemas/src/session-artifact.ts`
- `packages/schemas/src/report.ts`
- `packages/core/src/session/basic-session-runner.ts`
- `packages/core/src/session/session-store.ts`
- `apps/cli/src/main.ts`
- `tests/integration/core/`

Acceptance criteria:

- one interactive session produces one inspectable artifact containing multiple turns
- `/exit` closes the session cleanly and writes the artifact
- end-of-input also closes the session without corrupting reporting or persistence
- session inspection remains readable for interactive artifacts

### P2: Milestone Tests, Benchmark, And Docs

Goal:

- make `0.5.0` reviewable and stable under repeat execution

Dependencies:

- depends on all P0 and P1 work

Tasks:

- [x] add unit coverage for interactive command parsing and control-command handling
- [x] add integration coverage for multi-turn session continuity and exit behavior
- [x] add e2e coverage for at least one interactive two-turn repository task
- [x] add benchmark coverage for `interactive-multi-turn`
- [x] update architecture docs where the execution model or runtime flow changed
- [x] update release docs to reflect execution status as tasks land

Suggested files:

- `tests/unit/cli/`
- `tests/integration/core/`
- `tests/e2e/cli/`
- `benchmarks/interactive-multi-turn/`
- `docs/architecture/execution-model.md`
- `docs/architecture/overview.md`
- `docs/roadmap/releases/0.5.0.md`

Acceptance criteria:

- automated coverage exists for the new interactive command surface and multi-turn runtime behavior
- the milestone benchmark passes under repeated runs
- docs describe the new interactive session model without conflicting with prior one-shot CLI documentation

## Dependency Summary

1. `P0: Interactive CLI Contract And Command Surface`
2. `P0: Multi-Turn Session Runtime`
3. `P1: Interactive Rendering And Session Status`
4. `P1: Session Artifact And Exit Semantics`
5. `P2: Milestone Tests, Benchmark, And Docs`

## Scope Discipline

- [x] `0.5.0` does not introduce daemon mode or client/server session hosting
- [x] `0.5.0` does not require a full-screen TUI
- [x] `0.5.0` does not introduce multi-session orchestration in one process
- [x] `0.5.0` remains focused on foreground interactive CLI usability and multi-turn continuity

## Exit Rule

`0.5.0` is complete only when Code Orb can keep one interactive foreground session alive across multiple turns in a way that is understandable, inspectable, and useful for iterative local coding work.
