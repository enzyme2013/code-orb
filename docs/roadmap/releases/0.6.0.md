# Release 0.6.0

- Status: Planned

## Intent

Turn the current interactive CLI runtime into a more explicit and extensible execution layer without changing the core foreground CLI product shape.

`0.6.0` is where Code Orb should turn the tactical `0.5.x` stabilization patches into explicit runtime contracts.

The release is not about broadening the product surface yet. It is about removing hidden behavior in three places that would otherwise leak into future architecture:

- provider compatibility handling
- assistant-produced edit execution
- tool registration boundaries

`0.6.0` should also lay down the minimum explicit contracts that a real query loop and tool runtime will need next, without trying to land the full multi-iteration agent loop in the same milestone.

The goal is to keep the current CLI usable while making those paths deliberate enough to support later shells and external integrations without reworking the core runtime again.

## Why This Milestone Exists

Real `0.5.x` usage exposed two tactical paths that were acceptable for usability recovery but should not become long-term runtime shape:

- some OpenAI-compatible gateways only produced usable assistant text after a streaming recovery path
- general single-file coding requests could still depend on conversational code-block inference and filename heuristics before an edit was executed

`0.6.0` should absorb those lessons into documented runtime behavior instead of leaving them as accidental implementation details.

## In Scope

- define an explicit compatibility contract for the current OpenAI-compatible provider adapter:
  - supported response shapes
  - documented fallback behavior
  - documented unsupported or degraded behavior
  - capability differences that matter to core runtime correctness
- keep the runtime dependent on normalized provider capabilities and responses instead of provider-specific quirks
- define the minimum query-loop contract that is currently still implicit in the agent engine:
  - what counts as one loop iteration inside a turn
  - how deterministic execution, model output, tool execution, and verification feed back into turn state
  - what the explicit continue and stop conditions are
- replace the current heuristic generated-file path with an explicit execution contract for assistant-produced edits
- distinguish generated file creation, full-file rewrite, and targeted replacement as different runtime behaviors even if they share implementation pieces
- make generated edits auditable in the runtime rather than relying only on conversational code-block inference
- extract or define a tool registration boundary inside core so built-in tools are not the only possible registration shape
- define the minimum tool-runtime contract needed for later orchestration work:
  - registration vs dispatch
  - schema or input validation boundary
  - execution metadata expectations
  - declared timeout or interrupt behavior
- document any new architecture or contract surface introduced by the provider, generated-edit, or tool-registration work
- avoid introducing new CLI-specific assumptions while doing this work so later shell extraction remains possible

## Out Of Scope

- redesigning the interactive CLI command surface introduced in `0.5.0`
- background daemon mode or client/server orchestration
- a full plugin marketplace or user-facing extension ecosystem
- broad multi-provider routing or a large provider capability matrix
- model-native tool calling as a required dependency for this release
- a full generic multi-iteration query loop implementation
- concurrent tool orchestration, deferred tools, or streaming tool execution
- fully general multi-file autonomous editing
- general shell API extraction beyond what the contract cleanup above requires
- multi-agent execution as a default product path
- desktop UI implementation

## Exit Criteria

- provider behavior differences that matter to runtime correctness are documented through an explicit compatibility contract and reflected in adapter behavior
- fallback behavior between non-streaming and streaming provider paths is intentional, tested, and not just an undocumented recovery patch
- the runtime has an explicit documented query-loop contract even if the implementation remains intentionally simple
- the runtime no longer depends on a narrow heuristic single-file write path for common assistant-produced edits
- generated create, rewrite, and targeted replacement flows have an explicit execution contract and are distinguishable in reporting or events
- external or future tools can be registered behind a documented core boundary without changing current shell semantics
- the runtime has a documented tool registration and execution contract that future orchestration work can build on without redefining built-in tool behavior
- any change to extension boundaries or runtime protocol shape lands with the corresponding ADR or architecture updates in the same release
- the resulting runtime still supports the existing CLI flows without regressing `orb run`, `orb chat`, session artifacts, or the `interactive-multi-turn` benchmark

## Working Focus

`0.6.0` should favor explicit contracts over tactical patches.

This milestone should pay down specific extensibility debt from `0.5.x`, not become a stealth desktop milestone or a plugin-platform milestone.

The milestone question is:

- can Code Orb keep the current interactive CLI usable while making provider compatibility, generated-edit execution, and tool registration deliberate, testable, and extensible?

## Suggested Task Packages

### 1. Provider Compatibility Contract

- document the supported OpenAI-compatible response shapes and recovery behavior
- decide which differences stay adapter-local and which must surface as capabilities or explicit failures

### 2. Generated Edit Execution Contract

- replace code-block-and-path heuristics as the primary runtime contract for assistant-produced edits
- make create, rewrite, and replacement flows explicit enough to test and report clearly

### 3. Tool Registration Boundary

- separate built-in tool registration from built-in tool execution
- keep the boundary runtime-owned rather than CLI-owned

### 4. Query Loop And Tool Runtime Foundations

- document the explicit turn loop states and stop conditions that are currently implicit
- document the minimum tool-runtime contract that `0.7.0` can extend without another conceptual reset

### 5. Validation And Compatibility

- preserve current `orb run` and `orb chat` usability
- keep the interactive benchmark and session artifacts coherent after the contract changes

## Validation Strategy

### Unit Test

- provider response normalization and fallback behavior
- query-loop contract behavior at the current runtime level
- generated edit contract handling for create, rewrite, and replacement paths
- tool registration and unknown-tool behavior

### Integration Test

- interactive and one-shot flows still execute assistant-produced edits through the explicit contract
- provider recovery paths still return usable assistant output without changing core runtime semantics
- built-in tool behavior remains intact after registration boundary cleanup
- the explicit query-loop and tool-runtime contract still describes the real runtime behavior after the refactor

### End-To-End Or Benchmark

- `interactive-multi-turn` still passes
- the existing session artifact and reporting flows remain readable after the contract changes
