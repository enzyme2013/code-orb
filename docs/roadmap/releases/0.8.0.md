# Release 0.8.0

- Status: Planned

## Intent

Move Code Orb from an experimental CLI demo toward a CLI that is genuinely usable for real local-repository work.

`0.8.0` is the milestone where the runtime and shell should become reliable enough that contributors can use Code Orb for a meaningful subset of day-to-day coding tasks without treating every run as a fragile demo.

This is still not the same thing as a fully stable `1.0.0` CLI contract. The goal is usable and dependable, not frozen.

## In Scope

- harden the end-to-end CLI task loop for common local-repository work:
  - inspect
  - plan
  - edit
  - verify
  - report
- improve success and recovery behavior for common coding tasks after the `0.7.0` runtime skeleton, tool orchestration boundary, and provider adapter baseline are stable
- reduce the number of task classes that still depend on narrow special cases or fragile prompting
- improve error reporting and recovery guidance when provider, tool, verification, or approval paths fail
- make approval behavior and mutating-action reporting clearer during real CLI use
- support repository-level agent instructions well enough for real project use:
  - discover and read `AGENTS.md` or the authoritative project instruction surface
  - inject that guidance into execution predictably
  - make instruction use visible enough that users can tell the runtime is operating with project context
- expand benchmark and fixture coverage so CLI usability is measured against more realistic tasks than the current demo path
- tighten session and turn reporting so users can understand what happened without reading raw artifacts directly

## Out Of Scope

- a major TUI redesign or terminal UX overhaul
- desktop shells or desktop UI implementation
- plugin, MCP, or external extension ecosystems as product priorities
- broad multi-provider routing as a product feature
- background daemon mode
- core runtime layering changes that belong to `0.7.0`
- a stable `1.0.0` public CLI contract

## Exit Criteria

- Code Orb can complete a meaningful set of local-repository coding tasks with predictable inspect, edit, verify, and report behavior
- common failure paths produce actionable reporting instead of silent stalls or opaque errors
- approval and mutation reporting are understandable during normal CLI use
- repository instruction handling is reliable enough that Code Orb does not behave like a blank-slate agent on a guided project
- benchmark and test coverage demonstrate reliability beyond the current narrow demo path
- `orb run`, `orb chat`, session artifacts, and the current interactive benchmark remain functional
- the project can honestly describe the CLI as usable for real local work, even if the public contract is not yet `1.0.0` stable

## Working Focus

`0.8.0` should prioritize trustworthiness and operator confidence over visual polish.

It should assume the `0.7.0` runtime skeleton is already explicit enough that `0.8.0` can focus on reliability instead of reopening provider transport or session-versus-turn architecture.

The milestone question is:

- can a contributor reach for Code Orb for real local coding tasks without mentally classifying it as a demo every time?

## Suggested Task Packages

### 1. Common Task Reliability

- harden the paths that should work repeatedly in ordinary repository tasks
- reduce hidden task-specific fragility

### 2. Error And Recovery Quality

- improve failure reporting
- improve repair and fallback behavior where the runtime can act safely, on top of the adapter-owned fallback rules already established in `0.7.0`

### 3. Approval And Reporting Clarity

- make mutating actions and their outcomes easier to understand in real-time and in saved artifacts

### 4. Project Instructions

- make repository guidance such as `AGENTS.md` part of normal CLI execution rather than a manual workaround

### 5. Benchmark And Fixture Expansion

- validate usability against a broader set of realistic local tasks

## Validation Strategy

### Unit Test

- error classification and reporting paths
- approval and mutation-reporting behavior
- project-instruction discovery and injection behavior
- common task-shape helpers that remain in the runtime

### Integration Test

- realistic local task flows across inspect, edit, verify, and report
- recovery behavior after provider, tool, or verification failures
- repository-guided task flows that depend on project instructions being honored

### End-To-End Or Benchmark

- existing benchmark coverage still passes
- additional milestone benchmarks represent real CLI usage rather than only demo-style paths
