# Release 0.4.0 Todo List

## Purpose

This document is the ordered implementation plan for `0.4.0`.

It translates [0.4.0.md](./0.4.0.md) into concrete engineering stages so work can proceed without scope drift.

## Planning Assumptions

- keep the single-process CLI model from `0.3.0`
- prioritize local inspectable artifacts over remote/session-sync features
- treat git awareness as contextual reporting, not as commit automation
- keep follow-up support minimal: load prior session context and continue one new run intentionally

## Ordered Todo List

Follow this order unless there is a strong reason to change it:

1. lock contract and architecture updates for session artifacts and repository snapshots
2. persist per-run session artifacts reliably
3. add session store primitives and CLI inspection commands
4. add git snapshot collection and existing-change classification
5. wire follow-up context loading into `orb run`
6. add milestone benchmarks and stabilize tests
7. tighten release docs and validation baseline

## Stage 1: Contracts And Boundaries

- [x] Define schema-backed session artifact types in `packages/schemas` (artifact payload, snapshot payload, store index shape).
- [x] Extend report/event contracts only where needed so session and git context can be rendered without ad hoc fields.
- [x] Update architecture docs for execution model and event/report expectations related to persisted session artifacts.

Exit condition:

- contract shape for persisted sessions and git snapshots is explicit, versioned, and documented before runtime implementation

## Stage 2: Session Artifact Persistence

- [x] Add a session artifact writer in `packages/core` that persists one artifact per `orb run` (including failed or cancelled outcomes).
- [x] Ensure each artifact contains required `0.4.0` fields: task, cwd, timestamps, outcome, summary, changed files, validations, and risks.
- [x] Keep artifact writing auditable and deterministic enough for repeated benchmark runs.

Exit condition:

- one CLI run always produces a local inspectable artifact with required fields

## Stage 3: Session Store And Inspection

- [x] Add session store operations: save, list recent, load by id.
- [x] Add minimal CLI surfaces for inspection (for example: list sessions, show one session).
- [x] Keep command UX explicit and scriptable so benchmark automation can consume it.

Exit condition:

- users can list recent session records and load a specific record from CLI/runtime surfaces

## Stage 4: Git Snapshot And Change Classification

- [x] Capture git repository snapshot data when git is available: branch, dirty-before-run, staged, unstaged, untracked.
- [x] Capture snapshots at deterministic points (at least before run; optionally after run for clearer diff context).
- [x] Classify pre-existing repository changes versus changes caused by current run.
- [x] Define clear fallback behavior for non-git directories.

Exit condition:

- final report and stored artifact can distinguish prior changes from current-run changes for git-backed repos

## Stage 5: Follow-Up Context Flow

- [x] Add minimal follow-up input shape for `orb run` to reference a prior session record.
- [x] Build follow-up prompt/context payload from loaded session outcome (summary, changed files, validations, risks, unresolved items).
- [x] Ensure final reporting explicitly references follow-up context usage when present.

Exit condition:

- a second run can intentionally continue from a saved prior session rather than behaving as a blank slate

## Stage 6: Benchmarks And Test Matrix

- [x] Add `existing-changes-awareness` benchmark fixture and runner script.
- [x] Add `session-follow-up` benchmark fixture and runner script.
- [x] Add unit coverage for session serialization/store behavior, git snapshot parsing, change classification, and follow-up context builder.
- [x] Add integration/e2e tests for new CLI/session flows and keep existing `0.2.0`/`0.3.0` paths green.

Exit condition:

- both `0.4.0` milestone benchmarks pass consistently and regression tests remain green

## Stage 7: Release Gate And Documentation

- [x] Update roadmap/release docs with implementation status and final validation commands.
- [x] Record any cross-cutting protocol/architecture deltas in matching docs (`execution-model`, `events`, `tool-system`, `safety-model`) if changed.
- [x] Prepare `0.4.0` release assessment notes after benchmarks/tests are stable.

Exit condition:

- `0.4.0` has implementation-backed docs, reproducible validation, and clear release-readiness evidence

## Rule Of Order

Do not start benchmark broadening (Stage 6) before Stages 1 through 5 are implementation-stable.
